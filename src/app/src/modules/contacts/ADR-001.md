## Context and Problem Statement

The contacts module is the CRM backbone of AtendeAi. It must support:
- CRUD on contacts with tenant isolation
- Pipeline stage management with drag-and-drop
- Bulk operations (stage update, tagging, delete)
- CSV import with background job tracking
- Reports with async generation and download
- Cross-module integration with messaging (open conversation by contact)
- Mobile-responsive Brazilian SMB usage

## Decisions

### ADR-001-01: MVVM with ViewModel Hooks

**Decision**: Each page has a `useXxxViewModel` hook that owns state, side effects, and API calls. Views are thin.

**Rationale**: Separates presentation from logic without Redux overhead. ViewModels are unit-testable.

**Consequences**:
- (+) Clear ownership of business logic
- (+) No global store coupling
- (-) Prop drilling when multiple components need same VM slice
- (-) No cross-page state sharing without lifting

### ADR-001-02: TanStack Query for Server State

**Decision**: All API data flows through `useQuery` / `useMutation`. No client-side cache layer.

**Rationale**: Built-in cache invalidation, retries, stale-while-revalidate, optimistic updates.

**Consequences**:
- (+) Background refetch keeps data fresh
- (+) Loading/error states are declarative
- (-) Cache keys must be disciplined across modules
- (-) Mutation invalidation requires knowing all affected query keys

### ADR-001-03: Sheet (Drawer) over Modal for Forms

**Decision**: Create/Edit/Import/Reports all use side sheets, not center modals.

**Rationale**: Forms benefit from vertical space and persistent visibility of the underlying list.

**Consequences**:
- (+) Better for multi-field forms
- (+) Less disruptive than modals
- (-) Mobile: sheets can feel heavy if not collapsed to bottom-sheet

### ADR-001-04: Glass-card Aesthetic (CHALLENGED)

**Decision**: `glass-card` class applied to most surfaces with `backdrop-blur`, semi-transparent backgrounds, and `border-border/40`.

**Status**: **Contested by critique.** See P0 recommendations above.

**Rationale (original)**: Modern, differentiated from competitors.

**Critique**: Violates the "no decorative glassmorphism" design law. Multiple blurred surfaces on one page are visually noisy and GPU-expensive. Recommend reserving glass for one focal surface per page.

**Consequences**:
- (+) Visually distinctive on first impression
- (-) Performance cost on low-end devices
- (-) Reduces hierarchy clarity when everything is glass
- (-) Three+ background opacities (`/30`, `/60`, `/70`) create visual mud

### ADR-001-05: KPI Card Grid (CHALLENGED)

**Decision**: 4-up `KPICard` grid at top of list page showing Total / Pipeline / Customers / Inactive.

**Status**: **Contested by critique.**

**Critique**: Matches the "identical card grids" and "hero-metric template" bans. Recommend inline strip or single highlighted metric with trend.

### ADR-001-06: Tenant Isolation via API

**Decision**: All contacts-service endpoints include `tenantId` in path. No client-side tenant filtering.

**Rationale**: Security enforced server-side. Client never trusts tenant from body.

**Consequences**:
- (+) Cannot leak cross-tenant data from the client
- (-) Must verify every endpoint includes tenant in E2E tests

### ADR-001-07: Bulk Operations via Optimistic Updates

**Decision**: Bulk stage/tag/delete applies locally first, then syncs.

**Rationale**: Performance perception for large selections.

**Consequences**:
- (+) Instant feedback
- (-) Rollback complexity on partial failure
- (-) No undo affordance after success

## Alternatives Considered

| Alternative | Rejected Because |
|---|---|
| Redux Toolkit | Overkill for module-scoped state |
| MobX | Team familiarity lower |
| Component-only (no VM) | Logic scattered, harder to test |
| React Server Components | Rich client interactivity required |
| Modals for forms | Worse UX for multi-field inputs |
