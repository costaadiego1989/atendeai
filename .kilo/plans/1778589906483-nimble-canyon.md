## Contexto Levantado

### Stack Frontend (src/app)
- **Framework**: React 18 + TypeScript + Vite 5
- **State Management**: Zustand (auth-store, ui-store) + TanStack React Query (server state)
- **Routing**: React Router DOM v6
- **UI Library**: Radix UI + Tailwind CSS + shadcn/ui components
- **Forms**: React Hook Form + Zod validation
- **Charts**: Recharts
- **Testing**: Vitest (unit) + Playwright (E2E) — config existente mas quase vazio
- **Build**: Vite + SWC plugin

### Padrao Arquitetural Atual (MVVM)
```
modules/<nome>/
  ├── components/       # View (componentes React puros)
  ├── views/            # View (paginas/screens)
  ├── view-models/      # ViewModel (hooks com logica de estado/mutacoes)
  ├── services/         # Model (API clients, data access)
  └── utils/            # Helpers do modulo
```

### Modulos Frontend (20 modulos)
agent-rules, alerts, auth, billing, catalog, checkout, contacts, dashboard, inventory, messaging, platform-admin, proposals, prospecting, recovery, sales, scheduling, settings, social, support, users

### Camada Shared
- `shared/api/` — HTTP client com refresh token, error handling
- `shared/stores/` — Zustand stores (auth, ui)
- `shared/types/` — Tipos globais (User, Tenant, Contact, etc.)
- `shared/ui/` — Componentes reutilizaveis (Guards, Skeletons, KPICard, etc.)
- `shared/hooks/` — Hooks genericos (useUndo)
- `shared/services/` — Servicos utilitarios (CEP)
- `shared/commercial/` — Logica comercial compartilhada

## Deliverables

### DOC 1: `src/app/docs/FRONTEND-ARCHITECTURE.md`

#### Estrutura do Documento

1. **Visao Geral da Arquitetura**
   - Stack tecnologico
   - Diagrama de camadas (MVVM)
   - Fluxo de dados (API → Service → ViewModel → View)

2. **Analise SOLID**
   - S: Responsabilidade unica por modulo/componente
   - O: Extensibilidade via composicao (widgets dashboard, guards)
   - L: Substituicao (services abstraidos via interface implicita)
   - I: Segregacao (shared/types granular vs monolitico)
   - D: Inversao (apiClient como abstraction layer)

3. **Analise KISS / DRY / Clean Code**
   - Padroes repetidos a extrair (withBranchQuery, report jobs polling, CSV download)
   - Complexidade desnecessaria identificada
   - Nomenclatura e convencoes

4. **Analise MVVM**
   - Aderencia ao padrao por modulo
   - View-Models como hooks (padrao consistente)
   - Separacao de concerns (services nao importam React)

5. **Melhorias e Refactors Propostos**
   - P0: Tipos compartilhados monoliticos → split por dominio
   - P0: Padrao de report jobs (polling + download) → hook generico
   - P1: Error handling centralizado com domain exceptions
   - P1: Testes unitarios para view-models (cobertura atual: ~14 arquivos)
   - P1: Playwright config real (substituir lovable-agent placeholder)
   - P2: Lazy loading de modulos (code splitting por rota)
   - P2: MSW handlers compartilhados para testes
   - P3: Storybook para componentes shared/ui

6. **Gap Analysis: Frontend vs API**
   - Tabela modulo-a-modulo com status
   - Endpoints da API sem cobertura no frontend
   - Features da API Wave 1-4 que impactam frontend

7. **Roadmap de Implementacao**
   - Wave F1: Fechar lacunas P0
   - Wave F2: Refactors estruturais
   - Wave F3: Cobertura de testes
   - Wave F4: Polish e performance

## Decisoes Tecnicas

| Decisao | Escolha | Justificativa |
|---------|---------|---------------|
| Playwright config | Config propria (sem lovable-agent) | O wrapper atual e placeholder sem customizacao real |
| Auth strategy E2E | Storage state (cookie persistence) | Evita login em cada teste, alinha com cookie-based auth da API |
| Mock strategy | MSW (browser) para unit/integration, route intercept para E2E | MSW ja referenciado nos IMPLEMENTATION-GAP como padrao esperado |
| Idioma docs | Portugues (BR) | Alinhado com o projeto e equipe |
| Formato cenarios | Given/When/Then | Padrao BDD legivel para QA e devs |

## Riscos e Mitigacoes

| Risco | Mitigacao |
|-------|-----------|
| API endpoints mudam apos Wave 2-4 | Docs referenciam IMPLEMENTATION-GAP IDs; atualizar quando API mudar |
| Playwright config lovable-agent pode ter side effects | Remover dependencia e criar config from scratch |
| Testes E2E frageis sem backend real | Usar route intercept no Playwright para isolar frontend |
| Documentacao desatualizada rapidamente | Vincular a CI (lint check de links mortos) |
